# Minecraft Java Edition 개발 산출물 버전 호환·배포 범위

> 대상 버전: **1.7.10 ~ 26.2**  
> 대상 산출물: **모드 / 서버 플러그인 / 데이터팩**  
> 문서 목적: 같은 소스가 아니라, **동일한 배포 파일(JAR 또는 ZIP)을 여러 게임 버전에 제공해도 되는 범위**를 판단한다.  
> 최종 갱신 기준: **Minecraft Java Edition 26.2**

---

## 0. 먼저 읽을 것

이 문서에서 말하는 호환은 다음 세 개를 구분한다.

1. **동일 소스 호환**: 조건부 코드나 별도 소스셋을 사용해 한 저장소에서 관리할 수 있음
2. **동일 빌드 호환**: 같은 소스를 버전별로 다시 컴파일하면 됨
3. **동일 파일 호환**: 한 번 빌드한 동일한 JAR/ZIP을 여러 Minecraft 버전에 그대로 배포할 수 있음

배포 사이트에서 여러 게임 버전을 하나의 파일에 연결할 때는 **3번, 동일 파일 호환**만 기준으로 삼아야 한다.

### 판정 기호

| 기호 | 의미 |
|---|---|
| ✅ | 동일 파일 배포가 일반적으로 가능함 |
| ⚠️ | 조건부 가능. 사용 API·명령·레지스트리와 실제 실행 테스트가 필요함 |
| 🔧 | 소스 공유는 가능하지만 버전별 별도 빌드가 필요함 |
| ❌ | 동일 파일로 묶으면 안 됨 |
| — | 해당 기능 또는 플랫폼이 존재하지 않음 |

### 가장 중요한 원칙

- **모드:** 별도 근거가 없으면 Minecraft 버전별로 분리한다.
- **플러그인:** Bukkit 계층의 안정된 API만 사용했다면 여러 버전을 한 JAR로 지원할 수 있다.
- **NMS 플러그인:** 서버 내부 구현에 묶이므로 기본적으로 버전별 분리한다.
- **데이터팩:** `pack_format` 또는 데이터팩 버전이 같아도 명령어·JSON 스키마·레지스트리가 달라졌다면 자동 호환으로 보지 않는다.
- “서버가 JAR을 로드했다”와 “모든 기능이 정상 동작한다”는 같은 뜻이 아니다.
- 배포 범위의 양 끝 버전과 주요 중간 버전에서 자동 또는 수동 테스트하지 않았다면 공식 지원으로 표시하지 않는다.

# 2. 모드

## 2.1 모드 공통 결론

### 배포 기본값

| 상황 | 동일 JAR |
|---|---:|
| 같은 Minecraft 정식 버전 + 호환되는 로더 패치 업데이트 | ⚠️ |
| 같은 기능 릴리스의 패치 버전 | ⚠️ |
| Minecraft 마이너/기능 릴리스가 다름 | ❌ |
| Java 요구 버전이 달라짐 | ❌ |
| Forge와 Fabric | ❌ |
| Forge와 NeoForge | ❌ |
| Fabric과 Quilt | ⚠️ |
| 클라이언트·서버 양쪽에서 동일 모드 JAR 사용 | ✅ 가능하나 환경 메타데이터와 코드 분리 필요 |
| 로더 비의존 라이브러리만 공유 | ✅ 소스/라이브러리 공유일 뿐 모드 JAR 공유는 아님 |

모드는 게임 클래스에 직접 연결되고, 매핑·Mixin 대상·레지스트리·네트워크 패킷·데이터 생성 형식에 영향을 받는다. 따라서 **“패치 버전이니 될 것”이라는 이유만으로 한 파일로 묶으면 안 된다.**

### 한 파일 배포를 허용할 최소 조건

- 모드 메타데이터가 대상 Minecraft 버전 범위를 허용한다.
- 대상 범위에서 사용하는 모든 클래스·메서드·필드가 존재한다.
- Mixin 대상과 injection point가 대상 범위 전체에서 일치한다.
- 레지스트리 등록과 네트워크 프로토콜이 동일하다.
- 공통 Java 바이트코드 레벨로 컴파일되어 있다.
- 범위 내 모든 정식 버전에서 실제 게임 부팅 및 주요 기능 테스트를 통과했다.

하나라도 만족하지 않으면 별도 파일로 분리한다.

### Forge 메타데이터 주의

`mods.toml` 또는 해당 세대의 모드 메타데이터에 넓은 `minecraft` 버전 범위를 적을 수 있다고 해서 바이너리가 실제로 호환되는 것은 아니다. 범위 선언은 **로더의 사전 차단을 완화하는 메타데이터**이지, API/ABI 호환 보증이 아니다.

Fabric 공식 포팅 문서는 **1.21.11 이하용 기존 모드는 26.1에서 동작하지 않으며 컴파일 전용 의존성으로도 사용할 수 없다고 명시**한다. 또한 26.1 포팅에서 Java 25 전환과 매핑/리매핑 관련 빌드 변경이 요구된다. 따라서 1.21.11과 26.1은 반드시 별도 빌드·별도 파일로 분리한다.

---

## 2.5 Quilt

| 상황 | 동일 Fabric JAR을 Quilt에 배포 |
|---|---:|
| Quilt가 제공하는 Fabric 호환 계층으로 해당 모드가 실제 동작함 | ⚠️ |
| Fabric Loader 전용 내부 API 사용 | ❌ |
| Fabric API 특정 구현·Mixin 순서에 의존 | ⚠️ 또는 ❌ |
| Quilt 전용 API/QSL 사용 | Fabric과 동일 파일 불가 |
| 두 로더를 모두 CI에서 실행 테스트 | 조건부 허용 |

배포 페이지에서 Fabric과 Quilt를 동시에 표시하려면 “Quilt가 Fabric 모드를 읽을 수 있다”는 일반론만으로 결정하지 말고, 실제 Quilt 환경에서 테스트해야 한다.

## 2.6 멀티로더 프로젝트

Architectury, MultiLoader 템플릿, Stonecutter 또는 자체 공통 모듈을 사용해도 최종 산출물은 보통 다음처럼 분리한다.

```text
mod-common-code
├─ mod-fabric-1.20.1.jar
├─ mod-forge-1.20.1.jar
├─ mod-neoforge-1.20.2.jar
└─ mod-fabric-1.21.1.jar
```

공통 소스가 많다는 이유로 서로 다른 로더용 JAR을 하나로 합치면 안 된다. 하나의 물리적 JAR에 여러 로더 메타데이터와 구현을 넣는 특수 구조도 가능하지만, 모든 로더에서 검증된 프로젝트가 아니라면 일반 배포 지침으로 권장하지 않는다.

# 3. 서버 플러그인

## 3.1 플러그인 공통 결론

플러그인은 모드보다 넓은 버전 범위를 한 JAR로 지원할 수 있다. 이유는 Bukkit API가 서버 내부 구현을 추상화하기 때문이다. 단, 이 장점은 **공개 API만 사용했을 때**에만 성립한다.

| 플러그인 구현 방식 | 동일 JAR 지원 범위 |
|---|---|
| 오래된 Bukkit 공통 API만 사용 | 여러 주요 버전 가능 |
| Spigot API만 사용 | 사용한 API가 존재하는 버전부터 최신까지 조건부 가능 |
| Paper 전용 API 사용 | 해당 API가 존재하고 바이너리 호환되는 Paper 범위 |
| NMS/CraftBukkit 클래스 직접 참조 | 버전별 분리 |
| 리플렉션 NMS | 버전별 어댑터가 있고 모두 테스트된 경우만 한 JAR |
| ProtocolLib 사용 | ProtocolLib의 대상 버전 지원 여부까지 포함해 검증 |
| Folia 비호환 스케줄링 사용 | Folia 지원으로 표시 금지 |
| Java 8 바이트코드 | 신버전 서버에서 실행될 수 있으나 API 호환은 별도 문제 |
| Java 21/25 바이트코드 | 낮은 Java 서버에서 실행 불가 |

---

## 3.2 Bukkit / Spigot / Paper 공통 API 플러그인

### 권장 배포 묶음

아래 표는 **NMS 없음, CraftBukkit 없음, 버전 고유 enum/메서드 직접 참조 없음**을 전제로 한다.

| 지원 범위 | 동일 JAR 판정 | 조건 |
|---|---:|---|
| 1.7.10 ~ 1.8.9 | ⚠️ | 구형 Bukkit API 공통 부분만 사용하고 Java 8로 빌드 |
| 1.8.8/1.8.9 ~ 1.12.2 | ⚠️ | Material·Sound·EntityType 등의 버전 차이를 회피 |
| 1.13 ~ 1.16.5 | ⚠️ | Flattening 이후 API 기준. 1.12 이하와 같은 코드 경로로 묶지 않는 편이 안전 |
| 1.17 ~ 1.20.4 | ⚠️ | Java 바이트코드 및 신규 API 참조 관리 |
| 1.20.5 ~ 1.21.11 | ⚠️ | Java 21 서버 세대. ItemStack/컴포넌트 관련 API 사용 시 범위 축소 |
| 26.1 ~ 26.2 | ⚠️ | Java 25 계열과 최신 API 상태를 확인하고 별도 최신 빌드 권장 |
| 1.7.10 ~ 26.2 전체 | 이론상 가능하지만 비권장 | 극히 작은 공통 API만 사용하고 다중 버전 CI가 있을 때만 |

### 실무 권장 배포선

하나의 범용 플러그인을 운영할 때는 아래처럼 나누는 편이 안전하다.

```text
legacy-1.7.10-1.12.2.jar
flattened-1.13-1.16.5.jar
modern-java17-1.17-1.20.4.jar
modern-java21-1.20.5-1.21.11.jar
modern-java25-26.1-26.2.jar
```

이는 Minecraft API 경계뿐 아니라 Java 바이트코드 호환성을 분리하기 위한 권장안이다. 매우 보수적인 API만 사용하면 파일 수를 줄일 수 있지만, 그 경우에도 각 범위 테스트는 생략할 수 없다.

---

## 3.3 `api-version`

`plugin.yml`의 `api-version`은 플러그인이 어떤 Bukkit API 세대를 대상으로 작성되었는지 서버에 알린다.

### 규칙

- `api-version`보다 오래된 서버에서는 플러그인이 거부되거나 정상 로드되지 않을 수 있다.
- 최신 값을 적고 구버전도 지원한다고 표시하면 안 된다.
- 구버전까지 지원해야 한다면 지원하는 가장 오래된 API 세대에 맞춰 코드를 제한하거나, 빌드를 분리한다.
- `api-version`은 NMS 호환성을 해결하지 않는다.
- `api-version`을 생략해 레거시 변환 기능에 의존하는 방식은 장기적인 범용 배포 전략으로 권장하지 않는다.
- Paper에는 오래된 API 호환 처리를 비활성화하는 설정도 있으므로, “현재 Paper에서 우연히 로드됨”을 영구 보장으로 취급하지 않는다.

---

## 3.4 NMS 플러그인

### 기본 배포 단위

| NMS 대상 범위 | 동일 JAR |
|---|---:|
| 1.7.10 ↔ 1.8.x | ❌ |
| 1.8.x 내부 revision 차이 | ❌ 또는 버전별 어댑터 필요 |
| 1.9.x 내부 revision 차이 | ❌ 또는 버전별 어댑터 필요 |
| 서로 다른 주요/기능 릴리스 | ❌ |
| 1.20.4 ↔ 1.20.5 | ❌ |
| 1.21.11 ↔ 26.1 | ❌ |
| Paper 1.20.5 이전 ↔ 이후 매핑 방식 | ❌ 또는 재빌드·재검증 필요 |

Paper의 개발 문서에 따르면 Spigot 및 1.20.5 이전 Paper는 Spigot 매핑 계열을 사용하며, 최신 Paper의 개발·리매핑 환경과 차이가 있다. 리플렉션은 자동 리매핑 대상이 아니므로 문자열 클래스명·메서드명을 직접 관리해야 한다.

### 한 JAR에 여러 NMS 버전을 넣는 경우

다음 구조는 가능하다.

```text
nms/
├─ v1_8_R3/
├─ v1_12_R1/
├─ v1_16_R3/
├─ v1_20_R1/
└─ modern/
```

그러나 이 경우 “같은 파일이 자연스럽게 호환된다”는 의미가 아니다. **여러 버전 구현을 하나의 JAR에 번들한 멀티버전 플러그인**이다. 포함한 모든 구현을 테스트해야 한다.

---

## 3.5 Paper 전용 플러그인

| 사용 기능 | 배포 범위 |
|---|---|
| Bukkit API만 사용하고 Paper에서 실행 | Bukkit 플러그인 규칙 적용 |
| Paper API의 안정된 공통 기능 | API 도입 버전부터 조건부 지원 |
| 실험적 Paper Plugin API (`paper-plugin.yml`, bootstrapper/loader 등) | 버전별 또는 좁은 범위로 배포 |
| 데이터 컴포넌트 API | 1.20.5 전후를 분리 |
| 서버 내부 구현 또는 paperweight userdev 사용 | 빌드 대상 버전별 검증·분리 |

Paper 플러그인 시스템의 일부 기능은 실험적으로 문서화되어 있으므로, 넓은 버전 범위의 단일 JAR을 기본값으로 삼지 않는다.

---

## 3.6 Folia

Folia 지원 표시는 단순히 Paper에서 로드되는지로 판단하지 않는다.

- 전역 메인 스레드를 전제로 한 스케줄링을 사용하면 Folia 호환이 아니다.
- Region Scheduler, Entity Scheduler, Global Region Scheduler 등 적절한 실행 모델을 사용해야 한다.
- Paper와 Folia를 같은 JAR로 지원할 수는 있지만 플랫폼 감지와 스케줄러 추상화가 필요하다.
- Folia에서 실제 부하 테스트를 하지 않았다면 Folia 지원으로 표시하지 않는다.

---

## 3.7 프록시 플러그인

| 플랫폼 | 동일 파일 공유 |
|---|---:|
| BungeeCord ↔ Waterfall | ⚠️ API 공통 부분만 사용할 때 |
| BungeeCord/Waterfall ↔ Velocity | ❌ |
| Bukkit/Paper ↔ BungeeCord | ❌ |
| Bukkit/Paper ↔ Velocity | ❌ |
| 하나의 ZIP 안에 플랫폼별 JAR 포함 | ✅ 가능하지만 각 JAR은 별도 산출물 |

Velocity는 BungeeCord 플러그인 API와 같은 플랫폼이 아니므로 동일 플러그인 JAR로 묶지 않는다.

# 4. 데이터팩

## 4.1 데이터팩 존재 범위

| Minecraft 범위 | 데이터팩 |
|---|---:|
| 1.7.10 ~ 1.12.2 | — 정식 데이터팩 시스템 없음 |
| 1.13 ~ 26.2 | ✅ |

1.12.2 이하의 함수·전리품·발전 과제 관련 커스텀 콘텐츠와 1.13 이후 데이터팩은 같은 배포 형식이 아니다. 1.12.2 이하를 데이터팩 지원 버전으로 표시하면 안 된다.

---

## 4.2 데이터팩 동일 ZIP 판정법

다음 순서로 판정한다.

1. 대상 버전이 데이터팩을 지원하는가?
2. `pack.mcmeta`가 대상 범위를 표현할 수 있는가?
3. 폴더명이 대상 버전 전체에서 유효한가?
4. 모든 명령어 문법이 동일한가?
5. NBT·데이터 컴포넌트·아이템 문법이 동일한가?
6. recipe, loot table, predicate, advancement, tag, worldgen JSON 스키마가 동일한가?
7. 사용한 레지스트리 ID와 태그가 모두 존재하는가?
8. 양 끝과 중간 버전에서 `/reload` 오류가 없는가?
9. 모든 함수와 주요 동작 테스트를 통과했는가?

`pack_format` 경고가 없다는 이유만으로 4~9번을 생략하면 안 된다.

## 4.4 반드시 분리해야 하는 대표 경계

### 1.12.2 → 1.13

- 1.12.2 이하는 정식 데이터팩 시스템이 없다.
- 1.13에서 Flattening과 Brigadier 기반 명령 체계가 도입되었다.
- 동일 ZIP 배포 불가.

### 1.20.4 → 1.20.5

- 게임이 Java 21로 전환되었다.
- 데이터팩 버전이 41로 변경되었다.
- 아이템 데이터가 기존 NBT 중심 표현에서 데이터 컴포넌트 중심으로 크게 전환되었다.
- 아이템 명령·전리품·조건자·레시피·함수에서 관련 문법을 사용한다면 반드시 별도 파일로 분리한다.
- 단순 함수 팩이라도 양쪽 문법을 모두 검증하지 않았다면 묶지 않는다.

### 1.21.11 → 26.1

- 버전명 체계뿐 아니라 모딩 도구·Java 요구 버전·내부 데이터가 크게 바뀐 경계다.
- 데이터팩도 별도 대상으로 취급한다.
- `supported_formats`만 넓게 설정해 한 ZIP으로 선언하지 않는다.

### 26.1 → 26.2

- 26.2 정식판의 데이터팩 버전은 **107.1**이다.
- 26.1용 팩을 26.2에도 연결하려면 `pack.mcmeta`뿐 아니라 명령·JSON·레지스트리 차이를 검사해야 한다.
- 검증되지 않은 경우 별도 ZIP으로 배포한다.

---

## 4.5 `supported_formats` 사용 원칙

지원 범위를 메타데이터에 표현할 수 있는 버전에서는 하나의 팩이 여러 형식을 지원한다고 선언할 수 있다. 그러나 다음을 지킨다.

```json
{
  "pack": {
    "description": "Example",
    "pack_format": 0,
    "supported_formats": {
      "min_inclusive": 0,
      "max_inclusive": 0
    }
  }
}
```

위 값은 예시 자리표시자다. 실제 배포 시 대상 버전의 정확한 데이터팩 버전을 사용한다.

- 범위를 넓게 적는 것은 호환성 구현이 아니다.
- 게임이 팩을 선택 가능하게 만드는 것과 콘텐츠가 정상 동작하는 것은 별개다.
- 범위 안에서 제거·이름 변경된 레지스트리, 명령, JSON 필드가 하나라도 있으면 조건부 코드 또는 별도 팩이 필요하다.
- 오버레이 기능을 사용할 수 있는 세대라면 버전별 디렉터리로 차이를 흡수할 수 있지만, 이 역시 하나의 ZIP 안에 **버전별 구현을 함께 넣는 방식**이다.

# 6. 배포 전 체크리스트

## 6.1 모드

- [ ] 로더가 동일한가?
- [ ] `fabric.mod.json`, `mods.toml`, `neoforge.mods.toml`의 Minecraft 범위가 정확한가?
- [ ] Java 바이트코드 버전이 대상 전부에서 실행 가능한가?
- [ ] Mixin 대상과 injection point를 전 버전에서 검사했는가?
- [ ] Fabric API / Forge / NeoForge 의존성 범위가 맞는가?
- [ ] 클라이언트 전용 클래스를 전용 서버에서 로드하지 않는가?
- [ ] 네트워크 채널과 패킷 등록이 대상 전부에서 같은가?
- [ ] 최소·최대 및 중간 게임 버전에서 부팅·월드 진입 테스트를 했는가?

## 6.2 플러그인

- [ ] NMS 또는 CraftBukkit 클래스를 직접 참조하지 않는가?
- [ ] 최신 버전에만 존재하는 enum 상수를 정적 참조하지 않는가?
- [ ] `api-version`이 최소 지원 서버보다 높지 않은가?
- [ ] 컴파일한 Java 버전이 최소 지원 서버와 맞는가?
- [ ] Paper 전용 API를 Bukkit/Spigot 지원 범위에 포함하지 않았는가?
- [ ] Folia 지원 시 스케줄러 모델을 분리했는가?
- [ ] 최소·최대 서버에서 활성화, 명령, 이벤트, 저장/로드 테스트를 했는가?

## 6.3 데이터팩

- [ ] 정확한 데이터팩 버전 또는 지원 범위를 선언했는가?
- [ ] `/reload` 후 오류가 없는가?
- [ ] 모든 함수가 파싱되는가?
- [ ] 모든 recipe, loot table, predicate, advancement, tag가 로드되는가?
- [ ] 아이템 NBT/컴포넌트 문법이 대상 전체에서 유효한가?
- [ ] 제거되거나 이름이 바뀐 레지스트리 ID가 없는가?
- [ ] 최소·최대 및 중간 버전에서 주요 기능을 실행했는가?
- [ ] 테스트하지 않은 버전을 배포 사이트의 지원 목록에서 제외했는가?

# 7. 최종 판정 규칙

다음 의사결정 규칙을 그대로 적용한다.

```text
Q1. 파일 형식 또는 로더가 다른가?
 └─ 예: 별도 파일

Q2. Java 최소 버전이 다른가?
 └─ 예: 별도 빌드 권장. 높은 바이트코드로 빌드했다면 반드시 별도 파일

Q3. 게임 내부 클래스, NMS, Mixin, 매핑을 사용하는가?
 └─ 예: 기본적으로 Minecraft 버전별 별도 파일

Q4. 공개 Bukkit API만 사용하는 플러그인인가?
 └─ 예: API 도입 버전과 Java 세대를 기준으로 넓은 범위 테스트 가능

Q5. 데이터팩의 명령·JSON·레지스트리 구조가 같은가?
 └─ 아니오: 별도 ZIP 또는 오버레이/버전별 구현

Q6. 범위 내 모든 대표 버전에서 테스트했는가?
 └─ 아니오: 같은 파일로 지원 표시하지 않음

Q7. 모두 통과했는가?
 └─ 예: 동일 파일로 묶어서 배포 가능
```
